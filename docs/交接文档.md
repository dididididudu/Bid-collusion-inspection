## 一、系统架构与核心流程

### 1.1 整体数据流

```
Java 后端（按 itemCode 循环调用）
  │
  │  POST /api/v1/collusive-check/items/analyze
  │  {batchId, projectId, checkMode, itemCode, companies[]}
  │
  ▼
collusive_check_api.py
  │
  ├─ 轻量维度（不进管线，秒级返回）
  │  ├─ FILE_CODE_SIMILAR       → PDF /ID[0] 提取 → 交叉比对
  │  ├─ EDITOR_SIGNER_SIMILAR   → PDF Creator/Producer → 交叉比对
  │  ├─ DOC_AUTHOR_SIMILAR      → PDF Author → 交叉比对
  │  ├─ SAME_BID_CONTACT_SIMILAR → 正则提取手机/邮箱/人名 → 交叉比对
  │  └─ SAME_bidderName_SIMILAR → 正则提取公司名 → 交叉比对
  │
  └─ 重型维度（走完整 6 阶段管线）
     ├─ TECH_BID_SIMILAR        → 仅技术标页
     └─ Business_BID_SIMILAR    → 仅商务标页
         │
         ▼
     Phase 0: SCAN — 扫描 PDF 目录，提取元数据
         │
         ▼
     Phase 1: EXTRACT — 多进程分块文本/图片提取（PyMuPDF）
         │
         ▼
     Phase 1.5: EMBED — 全局 SBERT embedding 编码（一次性）
         │
         ▼
     Phase 2: SELECT — LSH + 元数据 + 向量 + 图片四级候选筛选
         │
         ▼
     Phase 3: ANALYZE — 逐对精细分析（三阶段匹配 + 四层图片比对）
         │
         ▼
     Phase 4: CLUSTER — 相似聚类与报告编排
         │
         ▼
     Phase 5: REPORT — 输出 detection_report.json + PDF
```

### 1.2 调用模式

- **不是一次返回全部维度**：Java 后端按 itemCode 顺序循环调用，每次只传一个 itemCode
- **不是按公司逐个调用**：每次请求携带同一批次全部公司
- 推荐调用顺序：先跑轻量项预热缓存 → TECH_BID_SIMILAR → Business_BID_SIMILAR（复用文本/OCR/embedding 缓存）

### 1.3 关键设计决策

| 决策 | 选择 | 原因 |
| :--- | :--- | :--- |
| PDF 引擎 | PyMuPDF (fitz) | 比 pdfplumber 快 10 倍，支持图片提取 |
| OCR 引擎 | RapidOCR (ONNX Runtime) | CPU 轻量，避免 PaddleOCR 高内存占用 |
| 语义模型 | paraphrase-multilingual-MiniLM-L12-v2 | 384 维，中文支持好，速度快 |
| 文本匹配 | MinHash LSH (datasketch) | 支持海量段落快速预筛 |
| 缓存 | SQLite（无 Redis） | 降低运维复杂度，单机够用 |
| 并行 | ProcessPoolExecutor | Phase 1/3 CPU 密集，多进程优于多线程 |

## 二、环境配置与权限

### 2.1 环境类型

| 环境 | 用途 | 说明 |
| :--- | :--- | :--- |
| **开发环境** | 本地开发调试 | Windows 10+ / CentOS，Python 3.10.11 |
| **测试服务器** | 集成测试 | CentOS 7+，8 核 16GB 起 |
| **生产服务器** | 线上服务 | CentOS 7+，16 核 32GB 起，建议 GPU |

### 2.2 第三方服务依赖

| 服务 | 用途 | 说明 |
| :--- | :--- | :--- |
| HuggingFace (hf-mirror.com) | 下载 SBERT 模型 | 镜像加速，内网需提前下载离线 |
| ONNX Runtime | RapidOCR 推理 | 自动下载模型，约 50MB |
| （无） | 系统无外部 API 依赖 | 所有检测纯本地计算 |

> **注意**：账号密码不写入文档，通过加密渠道单独传递。

### 2.3 配置加载优先级（从低到高）

```
1. config.py 默认值           ← DetectionConfig dataclass
2. config.json                ← from_json() 加载
3. 环境变量                    ← _apply_env_overrides()
4. .env 文件                  ← python-dotenv 加载
```

**不要在 default.yaml / config.py 默认值中写真实密钥。**

## 三、代码结构深度说明

### 根目录核心文件

---

#### `collusive_check_api.py` ★ 核心

**职责**: FastAPI 应用入口（1341 行）。接收 Java 后端 HTTP 请求，分发到轻量检测引擎或重型检测管线，构建响应。

**内部结构**:
- `/api/v1/collusive-check/health` — 健康检查
- `POST /api/v1/collusive-check/items/analyze` — 提交检测（同步返回或返回 taskId）
- `GET /api/v1/collusive-check/items/{taskId}` — 轮询结果
- 轻量维度：在 API 进程内直接调用 `_extract_lightweight()` 等方法，秒级返回
- 重型维度：提交到 `AnalyzeExecutor` 线程池，后台运行管线，通过 taskId 轮询

**坑点**:
- 轻量检查不进管线，但重型检查走完整管线，两者代码路径完全不同
- 同一 batchId 的 PDF 下载缓存到 `batch_downloads/{batchId}/`，按 `COLLUSIVE_DOWNLOAD_RETENTION_DAYS` 保留；请求正在使用的 batch 会跳过清理
- 进程内 `SBERT_DEVICE=cpu` 是硬编码默认，GPU 需通过环境变量设置
- dev 模式（`python collusive_check_api.py`）使用 uvicorn 单进程，生产环境建议 systemd + gunicorn

---

#### `main.py` — CLI 批量检测入口

**职责**: 离线模式，接收输入目录、输出目录，直接运行管线输出报告。不启动 HTTP 服务。

**关键参数**:
- `--input` — PDF 输入目录
- `--output` — 输出目录
- `--gpu` — 启用 GPU
- `--diagnose` — 仅环境诊断，不检测
- `--offline` — 离线模式（禁止模型下载）
- `--dimensions` — 逗号分隔的检测维度列表

**坑点**:
- `--diagnose` 模式下不检查 input 参数，直接打印环境信息退出
- `--offline` 设置 `TRANSFORMERS_OFFLINE=1` + `HF_HUB_OFFLINE=1` 双重离线
- 预加载的 SBERT 模型通过 `set_model()` 注入 orchestrator，避免 Phase 1.5 重复加载
- `HF_ENDPOINT=https://hf-mirror.com` 硬编码在文件头，国内环境已配置镜像

---

#### `config.py` — 配置参数系统

**职责**: `DetectionConfig` dataclass，管理所有可调参数。支持环境变量覆盖和 `.env` 文件加载。

**关键逻辑**:
- `__post_init__()` 自动验证参数合法性
- `_apply_env_overrides()` 读取环境变量覆盖配置
- Worker 数自动限制不超过 `os.cpu_count()`
- `.env` 文件路径硬编码为项目根目录下 `.env`

**坑点**:
- Worker 数超 CPU 核心数时会**静默修正**（输出 info 日志），不会报错
- `SBERT_DEVICE=auto` 会自动检测 CUDA/MPS，但仅在 `__post_init__` 时检查一次
- 所有环境变量名与 dataclass 字段名全大写一致
- 新增配置项需要在 `DetectionConfig` 添加字段，否则 `from_json()` 会忽略

---

#### `data_structures.py` — 数据结构定义

**职责**: 所有数据类定义。包括 `BidFeature`（文档特征）、`MetadataFeature`、`EvidenceChain`（证据链）、`PairwiseResult`（对级结果）、`GlobalReport`（全局报告）等 20+ 个 dataclass。

**关键类**:

| 类 | 用途 | 备注 |
| :--- | :--- | :--- |
| `BidFeature` | 单文档完整特征 | text_content 在流式模式下为空（存 SQLite） |
| `MetadataFeature` | PDF 元数据 | author/creator/producer/file_id/time_bucket |
| `ParagraphMatch` | 段落匹配详情 | detection_method 标记方法来源 |
| `TextEvidence` | 文本雷同证据 | 含 paragraph_matches + continuous_clone_blocks |
| `ImageEvidence` | 图片雷同证据 | 四层检测结果（L1-L4） |
| `ContactEvidence` | 联系人/公司雷同 | common_mobiles/emails/companies/credit_codes |
| `MetadataGroup` | 元数据聚合组 | 避免 N(N-1)/2 冗余展示 |
| `GlobalReport` | 全局报告 | 含所有 PairwiseResult + MetadataGroup |

---

#### `scoring.py` — 风险评分引擎

**职责**: 接收 `PairwiseResult` 列表和 `BidFeature` 列表，生成 `GlobalReport`。计算风险分、聚类、生成文件画像。

**评分逻辑**:
- 风险分 0-100，按阈值分 HIGH(70+) / MEDIUM(40+) / LOW(15+) / NONE
- `MetadataGroup` 聚合：将 N×(N-1)/2 条冗余对聚合成一条组展示
- 单文档风险画像（`FileProfile`）：统计关联可疑数、最高风险等级

---

#### `report.py` — 报告生成器

**职责**: 将 `GlobalReport` 序列化为 `detection_report.json`。仅输出 JSON（PDF 报告由 ReportLab 生成，代码在内部）。

**坑点**:
- 历史版本输出过 PDF 报告，当前版本已不再默认生成 PDF（`detection_report.pdf` 是遗留产物）
- JSON 输出通过 `json.dumps(..., ensure_ascii=False, indent=2)` 保证中文可读

---

#### `test_api.py` — 综合测试框架

**职责**: 包含两个层面的测试：
1. **API 集成测试**（`--module health/validation/lightweight/heavy/error`）— 需启动 API 服务
2. **内部单元测试**（`--module internal`）— 无需 API 服务，直接测试内部模块

**内部测试覆盖**:
- 文本相似度（含位置随机化 10 轮 + 9 个标准用例）
- 文件码提取、作者提取、编辑提取、联系人提取、公司名提取、信用代码提取

---

#### `test_evidence_unit.py` — 证据单元测试

**职责**: Mock 方式测试 `_get_company_results_from_pipeline()` 的证据构建逻辑。测试 FAILED 状态下 paragraphs + images 证据格式，以及 SUCCESS 空证据。

**独特价值**: 不启动 API、不跑管线，直接 mock 结果验证 evidence 结构正确性。

---

### extraction/ — PDF 提取层

---

#### `extraction/pdf_extractor.py` ★ 核心

**职责**: 基于 PyMuPDF (fitz) 的 PDF 提取引擎。提取文本、元数据、嵌入图片、执行页级分类。

**核心方法**:
- `extract_metadata()` — Phase 0 调用，仅读取元数据
- `extract_chunks()` — Phase 1 调用，按 chunk 流式提取文本
- `extract_images_from_page()` — 提取单页嵌入图片
- `classify_pages()` — 技术标/商务标页级分类

**坑点**:
- **PyMuPDF 非线程安全**：`fitz.Document` 不能在线程间共享，多线程需每线程 `open()` 独立实例
- 提取嵌入图片用 `doc.extract_image(xref)`，但矢量图/碎片图需要通过 `cluster_drawings()` 检测区域后单独渲染
- **不做整页渲染 OCR**：仅提取嵌入图片 + 合并矢量区域渲染
- 每 10 页切一个 batch 控制内存

---

#### `extraction/text_processor.py` — 文本处理器

**职责**: 文本清洗、分块、段落分割、jieba 分词、MinHash 签名计算。

**处理流程**:
1. 原始文本 → 去除空白/特殊字符
2. jieba 分词 → 构建 MinHash 签名（128 维）
3. 段落分割 → 每段落独立签名
4. 技术标/商务标文本分别处理

**坑点**:
- 分词结果影响 MinHash 签名质量，jieba 词典需要覆盖投标领域术语
- `MINHASH_NUM_HASHES=128` 硬编码，改大会显著增加内存和计算量

---

#### `extraction/contact_extractor.py` — 联系人提取

**职责**: 通过正则表达式从文档全文中提取手机号、邮箱、公司名称、统一社会信用代码、会员号。

**正则策略**:
- 手机号: `1[3-9]\d{9}`（11 位中国大陆手机号）
- 邮箱: 标准 email 正则
- 公司名: 基于后缀列表（有限公司、集团、股份等）+ 上下文检测
- 信用代码: 18 位统一社会信用代码正则

**坑点**:
- 纯正则匹配，不依赖 NLP，速度快但精度有限
- 公司名提取不是命名实体识别，可能漏检非常见后缀的公司名
- 手机号正则可能匹配到非手机号数字（需置信度过滤）

---

#### `extraction/feature_cache.py` — SQLite 缓存

**职责**: `DocumentCache` 类，所有文本/段落/图片/Minhash/向量数据持久化到 SQLite。

**关键设计**:
- WAL 模式 + 批量提交优化写入
- zlib 压缩文本块（压缩比 5:1~8:1）
- 惰性加载：按需从 SQLite 读取文本
- SCHEMA_VERSION = 10 跟踪数据库结构变更

**坑点**:
- 多进程写冲突时自动重试（指数退避）
- `DISABLE_CACHE=1` 会在启动时删除 `cache/features.db*` 并重建
- 数据库文件锁在进程退出前不会释放

---

#### `extraction/base.py` — 提取器基类

**职责**: `BasePDFExtractor` 抽象基类，定义 `extract_metadata()` 和 `extract_chunks()` 接口。当前只有 PyMuPDF 实现，pdfplumber 为回退方案。

---

### pipeline/ — 管线编排（核心业务逻辑）

---

#### `pipeline/orchestrator.py` ★ 核心

**职责**: `BidDetectionOrchestrator` 类，管理六阶段管线的完整执行流程。协调 extractor、matcher、scoring 等所有模块。

**关键方法**:
- `detect(input_dir, output_dir)` — 主入口，按 Phase 0→1→1.5→2→3→4→5 执行
- `_run_phase_0()` / `_run_phase_1()` / ... — 各阶段实现
- `_get_company_results_from_pipeline()` — API 调用时提取公司级结果

**坑点**:
- Phase 1 并行提取时，**切勿**线程间共享 `fitz.Document`
- Phase 1.5 全局 SBERT 编码是单次阻塞操作，大文档可能耗时数分钟
- technical 和 commercial 的 Phase 3 checkpoint 分开保存
- TOC 过滤（目录段落不参与查重）默认启用，`TOC_FILTER_ENABLED` 控制

---

#### `pipeline/evidence_builder.py` — 证据构建

**职责**: 共享的证据构建函数，供 `orchestrator` 和 `parallel_workers` 共用。

**函数**:
- `build_metadata_evidence()` — 元数据比对
- `build_text_evidence()` — 文本证据拼装
- `build_contact_evidence()` — 联系人证据拼装

**坑点**: 确保 API 响应和 JSON 报告中的 evidence 结构一致，两处用同一份代码。

---

#### `pipeline/parallel_workers.py` — 并行 Worker

**职责**: 模块级函数，供 `ProcessPoolExecutor` 调用。每个 worker 创建自己的 SQLite 连接。

**函数**:
- `process_chunk_worker()` — Phase 1 chunk 提取
- `match_pair_worker()` — Phase 3 逐对分析
- `extract_metadata_worker()` — 元数据提取

**坑点**:
- 所有函数必须是模块级（可 pickle），不能是类方法
- `_begin_immediate_with_retry()` 处理 SQLite 写冲突，最多重试 10 次
- OCR 调用通过 `GPUManagerClient` 走独立进程（防止 GPU OOM）

---

#### `pipeline/checkpoint.py` — 断点续传

**职责**: `CheckpointManager`，每阶段写入 JSON checkpoint。Phase 3 每 N 对增量写入。

**存储结构**:
```json
{
  "phase": 3,
  "completed_pairs": 150,
  "total_pairs": 500,
  "config_hash": "sha256...",
  "version": 4
}
```

**坑点**:
- `config_hash` 检测配置漂移，配置变更会强制重新开始
- technical 和 commercial 的 checkpoint 独立存储

---

#### `pipeline/streaming_context.py` — 流式上下文

**职责**: `StreamingContext`，管理文本块的惰性加载。仅在需要时从 SQLite 读取文本，控制内存占用。

---

#### `pipeline/ocr_helpers.py` — OCR 辅助

**职责**: OCR 批处理辅助函数，包括 `ocr_pages()`（页级 OCR 调度）和 `aggregate_ocr_paragraphs()`（聚合识别结果）。

---

#### `pipeline/gpu_manager.py` — GPU 资源管理

**职责**: `GPUManager`（主进程）+ `GPUManagerClient`（worker 进程）。独立进程持有唯一 OCR 引擎，worker 通过队列提交图片批量推理。

**架构**:
```
GPUManager (主进程)          GPUManagerClient (worker 进程)
┌──────────────────┐        ┌─────────────────────┐
│  _run() 进程      │◄─队列─│  batch_ocr(images)   │
│  唯一 OCR 引擎     │──队列─►  阻塞等待结果          │
└──────────────────┘        └─────────────────────┘
```

**坑点**:
- 解决多进程各自创建 OCR 引擎导致 OOM 的问题
- 仅 GPU 场景生效，CPU 场景直接使用 `ImageOCREngine`

---

#### `pipeline/toc_parser.py` — 目录过滤

**职责**: 技术标/商务标页级分类器。按可靠性从高到低执行：目录条目页码 → 正文标题锚点 → 页级关键词密度兜底。

**关键词**: 预定义 `TECH_KEYWORDS` 列表（技术标、施工组织设计、技术方案等）和 `COMMERCIAL_KEYWORDS` 列表（商务标、报价等）。

**坑点**: strict 模式（`DIMENSION_STRICT_FILTER=True`）下分类失败不会回退全文混用，保证技术/商务接口互不污染。

---

### matching/ — 文本匹配层

---

#### `matching/paragraph_matcher.py` ★ 核心

**职责**: `ParagraphMatcher`，三阶段段落匹配引擎：
1. **Stage 1**: MinHash Jaccard 向量化批量计算（numpy 广播，100x 加速）
2. **Stage 2a**: 精确词级 Jaccard（基于 jieba 分词）
3. **Stage 2b**: SBERT 语义验证（查表，不调模型）

**坑点**:
- `MINHASH_NUM_HASHES=128` 硬编码，改大则索引重建变慢、内存翻倍
- `SEQUENCE_MATCHER_MAX_CHARS=1200` 长段落提前短路，避免 difflib 瓶颈
- Stage 1 返回 Top-K 候选再进入 Stage 2，`PARAGRAPH_MATCH_STAGE1_TOP_K=5000`

---

#### `matching/semantic_matcher.py` — SBERT 语义引擎

**职责**: SBERT embedding 推理。设备自动检测 CUDA → MPS → CPU。智能去重：同一段落只编码一次。

**坑点**:
- 首次加载模型耗时（下载约 400MB），`local_files_only=True` 可离线
- ONNX 加速可选（optimum.onnxruntime），但当前默认不使用
- 编码结果按 text_hash 全局缓存，跨批次复用

---

#### `matching/lsh_index.py` — LSH 索引

**职责**: datasketch MinHash LSH 索引封装。提供文档级和段落级两种索引。

**设计**:
- 文档级 LSH：用于 Phase 2 候选对初筛
- 段落级 LSH：按需临时构建，用完释放

**坑点**: 旧方案维护全局段落 LSH 索引（100K 段落 × 8 band = 80 万条目），新方案按需构建，每次仅约 200 条目。

---

#### `matching/selector.py` — 候选对选择器

**职责**: `CandidatePairSelector`，四级筛选策略：
1. **Level 1**: datasketch MinHashLSH 文档级筛选（O(n)）
2. **Level 2**: 元数据指纹匹配（software_fingerprint / time_bucket 相同 → 强制候选）
3. **Level 3**: 文档向量余弦预筛
4. **Level 4**: 图片哈希匹配

---

### embedding/ — 向量嵌入

---

#### `embedding/embedding_engine.py` — SBERT 编码引擎

**职责**: 全局段落 embedding 编码、缓存、查询。

**关键设计**:
- 按 text_hash 缓存 embedding，**跨批次复用**
- `NORMALIZE_EMBEDDINGS=True` 编码时归一化，后续余弦相似度直接点积
- Phase 3 预加载所有段落向量到内存（`PHASE3_PRELOAD_EMBEDDINGS=True`）

**坑点**: 预加载加快 Phase 3 速度但增加内存占用（每条向量 384 维 float32 = 1.5KB，10 万条 = 150MB）。

---

### image_analysis/ — 图片分析

---

#### `image_analysis/image_hasher.py` — 图片哈希

**职责**: 三级级联图片匹配：
1. **L1**: 多哈希共识（pHash + dHash + 长宽比）— 快速预筛
2. **L2**: ORB 特征匹配 — 确认结构一致性
3. **L3**: 直方图相关性 — 确认颜色一致性

**坑点**: 三级都通过才判为匹配，降低误报。ORB 检测器按 `nfeatures` 分级缓存。

---

#### `image_analysis/image_matcher.py` — 图片比对引擎

**职责**: 集成四层图片比对逻辑，构建 `ImageEvidence`。

---

#### `image_analysis/image_ocr.py` — OCR 引擎

**职责**: 包装 RapidOCR/PaddleOCR/EasyOCR 的统一接口。`ImageOCREngine` 提供 `ocr_image()` 和 `ocr_batch()`。

**坑点**:
- **首次加载模型耗时约 30 秒**（下载 ONNX 模型 + 初始化），已做懒加载，非卡死
- `OCR_WORKERS=1` 是 CPU 推荐值，多进程争 CPU 反而更慢
- GPU 场景下 `onnxruntime-gpu` 需额外安装，默认不依赖
- RapidOCR 模型约 50MB，自动下载到 `~/.rapidocr/`

## 四、核心业务流程详解

### 4.1 投标文件处理完整流程

```
1. 文件上传
   Java 后端传入 bidFileUrl
   → API 流式下载到 batch_downloads/{batchId}/{companyRecordId}_{bidderName}.pdf
   → 多文件并发下载（PDF_DOWNLOAD_WORKERS=4）

2. PDF 提取（Phase 0 + Phase 1）
   PyMuPDF 打开 → 检查加密 → 提取元数据（作者/creator/producer/文件码）
   → 逐页提取文本 → 每 10 页切一个 chunk
   → 嵌入图片直接 extract_image(xref)
   → 矢量图/碎片图 cluster_drawings() 检测区域 → 合并 bbox → 只渲染合并区域

3. 页级分类（Phase 1）
   目录条目页码 + 正文标题锚点 → 确定 technical/commercial 页范围
   关键词密度兜底 → 每页打标签
   strict 模式：分类失败不混用全文

4. OCR 识别（Phase 1）
   RapidOCR ONNX Runtime
   → 识别嵌入图片和渲染区域的文字
   → OCR 结果逐页缓存到 SQLite
   → 技术标/商务标分别存储

5. 文本向量化（Phase 1.5）
   jieba 分词 → MinHash 签名（128 维）
   SBERT embedding 编码（384 维）→ 按 text_hash 缓存

6. 候选筛选（Phase 2）
   LSH + 元数据指纹 + 文档向量 + 图片哈希 四级筛选
   → 输出候选对列表（上限 MAX_CANDIDATE_PAIRS=5000）

7. 逐对精细分析（Phase 3）
   Stage 1: MinHash Jaccard 向量化批量计算
   Stage 2a: 精确词级 Jaccard
   Stage 2b: SBERT 语义验证（查表点积，不调模型）
   图片 L1-L4 比对
   → 输出每对 PairwiseResult

8. 聚类与报告（Phase 4 + Phase 5）
   MetadataGroup 聚合 → 风险评分 → JSON 序列化
   → detection_report.json + 公司级 API 响应
```

### 4.2 缓存复用规则

| 缓存 | 位置 | 生命周期 | 复用条件 |
| :--- | :--- | :--- | :--- |
| PDF 下载 | `batch_downloads/{batchId}/` | 默认保留 30 天，`COLLUSIVE_DOWNLOAD_RETENTION_DAYS` 控制 | batchId 相同 |
| 工作目录 | `collusive_workdir/batch_*` | 默认保留 7 天，`COLLUSIVE_WORKDIR_RETENTION_DAYS` 控制 | batchId + 文件内容未变 |
| 文本特征 | `cache/features.db` | `DISABLE_CACHE` 控制 | doc_id 相同 |
| embedding | `cache/features.db` 中按 text_hash | 永久 | text_hash 相同 |
| checkpoint | `collusive_workdir/.../checkpoints/` | 阶段完成 | batch 未中断 |

### 4.3 内存与性能调优

```bash
# 8 核 CPU 推荐（均衡）
export PHASE1_WORKERS=2
export PDF_CHUNK_WORKERS=4
export OCR_COLLECT_WORKERS=4
export PHASE3_WORKERS=4

# 如果 OOM Killed，降低到（保守）
export PHASE1_WORKERS=1
export PDF_CHUNK_WORKERS=1
export OCR_COLLECT_WORKERS=1
export PHASE3_WORKERS=1
```

关键内存消耗点：
- SBERT embedding 预加载：N 条 × 1.5KB（384 维 float32）
- OCR 中间结果：RGB 图片解码后 ≈ 原图字节数
- Phase 3 段落缓存：全文段落文本 + 向量

### 4.4 缓存保留与健康检查

API 启动时执行一次缓存保留期扫描；请求进入时按 `COLLUSIVE_CLEANUP_INTERVAL_SECONDS` 控制扫描频率。正在执行的 batch 和当前请求 batch 不会被删除。

```bash
export COLLUSIVE_CLEANUP_ENABLED=1
export COLLUSIVE_CLEANUP_INTERVAL_SECONDS=3600
export COLLUSIVE_DOWNLOAD_RETENTION_DAYS=30
export COLLUSIVE_WORKDIR_RETENTION_DAYS=7
```

`COLLUSIVE_DOWNLOAD_RETENTION_DAYS` 或 `COLLUSIVE_WORKDIR_RETENTION_DAYS` 设置为 `0` 或负数时，对应目录不执行过期清理。`collusive_workdir/shared_embedding_cache` 不匹配 `batch_*`，不在工作目录清理范围内。

健康检查可以交给 systemd/crontab：

```bash
# 每 5 分钟健康检查
*/5 * * * * curl -sf http://localhost:8001/api/v1/collusive-check/health || systemctl restart bid-collusion-api
```

### 6.4 常见问题排查

| 现象 | 排查步骤 | 可能原因 |
| :--- | :--- | :--- |
| 健康检查 200 但检测返回 ERROR | 查看服务端日志中的异常堆栈 | PDF 加密/损坏/格式不支持 |
| 服务被 Killed | `dmesg \| tail -20` 查看 OOM Killer | worker 数太高/PDF 太大 |
| Phase 3 卡住 | 查看 checkpoint 进度文件 | SQLite 写锁死锁（等待超时） |
| OCR 结果为空 | `python -c "from rapidocr_onnxruntime import RapidOCR; RapidOCR()"` | 模型未下载/损坏 |
| SBERT 加载失败 | 检查 `./models/` 目录权限和磁盘空间 | 模型不完整/磁盘满 |
| 报告乱码 | 检查系统字体目录 | 缺少中文字体 |

### 6.5 重要配置项修改提醒

| 配置项 | 修改影响 | 建议 |
| :--- | :--- | :--- |
| `MINHASH_NUM_HASHES` | 改大 → LSH 索引重建变慢、内存翻倍；改小 → 精度下降 | **不建议修改** |
| `SBERT_BASE_THRESHOLD` | 改大 → 漏报增多；改小 → 误报增多 | 按业务需求微调（0.60-0.85） |
| `PHASE[1/3]_WORKERS` | 改大 → CPU 争抢，实际变慢 | 不超过 CPU 核心数/2 |
| `CHUNK_PAGE_SIZE` | 改大 → 单 chunk 内存增加；改小 → chunk 数增多 | 保持 50 |
| `DIMENSION_STRICT_FILTER` | 改 false → 技术/商务标内容混用 | **不建议改为 false** |
