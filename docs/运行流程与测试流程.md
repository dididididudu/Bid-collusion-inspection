# 围标串标 AI 服务运行与测试流程

本文档用于本地、CPU/GPU 服务器、Java 联调场景下测试 `collusive_check_api.py`。

## 1. 测试前准备

### 1.1 PDF 文件目录

测试 PDF 放在 `batch_downloads/{batchId}` 目录下，文件名必须能解析出公司记录 ID 和公司名：

```text
batch_downloads/75689/701_测试公司A.pdf
batch_downloads/75689/702_测试公司B.pdf
```

命名规则：

```text
{companyRecordId}_{bidderName}.pdf
```

注意：

- `batchId` 推荐直接使用目录名，例如 `75689`。
- `companyRecordId` 必须是数字。
- 至少需要 2 个 PDF。
- 如果只替换 PDF 内容但不换 `batchId`，需要删除旧缓存：

```bash
rm -rf collusive_workdir/batch_75689
```

Windows PowerShell：

```powershell
Remove-Item -Recurse -Force collusive_workdir\batch_75689
```

## 2. 启动 API 服务

在项目根目录启动：

```bash
python collusive_check_api.py
```

默认接口地址：

```text
http://127.0.0.1:8001
```

服务启动时会预加载 SBERT 和 OCR。首次启动较慢是正常现象。

## 3. 推荐测试方式：一键脚本

这是最推荐的测试方式。脚本会自动：

- 扫描 PDF 目录；
- 启动本地 HTTP 文件服务；
- 构造和 Java 后端一致的请求 JSON；
- 按 itemCode 顺序调用 API；
- 输出每个检查项的耗时和结果数量。

### 3.1 测试全部检查项

```bash
python scripts/test.py --pdf-dir batch_downloads/75689
```

等价于测试全部 7 项：

```text
FILE_CODE_SIMILAR
DOC_AUTHOR_SIMILAR
EDITOR_SIGNER_SIMILAR
SAME_BID_CONTACT_SIMILAR
SAME_bidderName_SIMILAR
TECH_BID_SIMILAR
Business_BID_SIMILAR
```

### 3.2 只测试轻量项

轻量项不跑 OCR/SBERT，主要用于快速验证接口、PDF 下载、元数据、联系人/公司名提取。

```bash
python scripts/test.py --pdf-dir batch_downloads/75689 --lightweight
```

包含：

```text
FILE_CODE_SIMILAR
DOC_AUTHOR_SIMILAR
EDITOR_SIGNER_SIMILAR
SAME_BID_CONTACT_SIMILAR
SAME_bidderName_SIMILAR
```

### 3.3 只测试重型项

重型项会跑文本相似、图片相似、OCR、SBERT，耗时较长。

```bash
python scripts/test.py --pdf-dir batch_downloads/75689 --heavy
```

包含：

```text
TECH_BID_SIMILAR
Business_BID_SIMILAR
```

### 3.4 测试单个检查项

```bash
python scripts/test.py --pdf-dir batch_downloads/75689 --item DOC_AUTHOR_SIMILAR
python scripts/test.py --pdf-dir batch_downloads/75689 --item TECH_BID_SIMILAR
python scripts/test.py --pdf-dir batch_downloads/75689 --item Business_BID_SIMILAR
```

### 3.5 输出怎么看

示例：

```text
[TECH_BID_SIMILAR] 125.3s  FAILED=2 SUCCESS=0 ERROR=0
```

含义：

```text
FAILED  = 检测不通过，发现雷同/异常
SUCCESS = 检测通过
ERROR   = 程序执行异常，查看 error_report.jsonl
```

## 4. Java 后端接口方式：生成请求 JSON + curl

这种方式用于验证 Java 后端实际调用的 HTTP 接口格式。

### 4.1 生成请求文件

```bash
python scripts/gen_test_payloads.py batch_downloads/75689
```

生成文件：

```text
batch_downloads/75689/req_FILE_CODE_SIMILAR.json
batch_downloads/75689/req_DOC_AUTHOR_SIMILAR.json
batch_downloads/75689/req_EDITOR_SIGNER_SIMILAR.json
batch_downloads/75689/req_SAME_BID_CONTACT_SIMILAR.json
batch_downloads/75689/req_SAME_bidderName_SIMILAR.json
batch_downloads/75689/req_TECH_BID_SIMILAR.json
batch_downloads/75689/req_Business_BID_SIMILAR.json
```

### 4.2 启动 PDF 文件服务

另开一个终端：

```bash
python -m http.server 18081 --directory batch_downloads/75689
```

为什么建议启动文件服务：

- `gen_test_payloads.py` 生成的 `bidFileUrl` 指向 `http://127.0.0.1:18081/...`。
- Windows 下脚本可能创建 `701_test.pdf` 这种 ASCII 文件名副本，HTTP 文件服务可以稳定访问。
- 这样最接近 Java 后端“通过 URL 传 PDF”的真实调用方式。

### 4.3 执行 curl

## 5. 本地文件复用规则

API 下载 PDF 前会先检查本地是否已有文件：

```text
batch_downloads/{batchId}/{companyRecordId}_{bidderName}.pdf
```

如果该文件存在且大小大于 100 字节，API 会跳过下载，直接使用本地文件。

这条规则的限制：

- 文件名必须和请求里的 `companyRecordId`、`bidderName` 完全匹配。
- 如果请求里 `bidderName` 是 `测试公司A`，本地文件必须类似 `701_测试公司A.pdf`。
- 如果本地只有 `701_test.pdf`，API 不会把它当成 `701_测试公司A.pdf` 的本地缓存。
- 如果本地文件名不匹配，就必须保证 `bidFileUrl` 可以访问，否则当前 `itemCode` 会返回全公司 `ERROR`。

因此：

- 使用 `scripts/test.py` 时，不需要手动管这些细节，脚本会自动启动文件服务。
- 手写 curl 时，推荐启动 `python -m http.server`，不要随意填写无效 `bidFileUrl`。

## 6. 缓存复用测试

同一 `batchId` 共享缓存：

```text
collusive_workdir/batch_{batchId}/cache/features.db
```

推荐顺序：

```bash
python scripts/test.py --pdf-dir batch_downloads/75689 --lightweight
python scripts/test.py --pdf-dir batch_downloads/75689 --item TECH_BID_SIMILAR
python scripts/test.py --pdf-dir batch_downloads/75689 --item Business_BID_SIMILAR
```

预期日志：

```text
文本缓存预热完成/复用
文档已完全提取
OCR: ... 已有 ... 条结果，跳过
Phase 1.5: 嵌入缓存命中 ...
```

缓存说明：

- 轻量项会预热文本、段落、联系人指纹缓存。
- 重型项会在同一缓存库上继续补 OCR、SBERT embedding、候选对、图片证据。
- 相同 `batchId` 重复运行会复用已有结果。
- 如果换了 PDF 内容，必须清理 `collusive_workdir/batch_{batchId}`。

## 7. 结果文件和异常排查

JSON 报告：

```text
collusive_workdir/batch_{batchId}/output/detection_report.json
```

异常报告：

```text
collusive_workdir/batch_{batchId}/output/error_report.jsonl
```

接口返回 `status=ERROR` 时，响应里会包含：

```json
{
  "error": "错误信息",
  "stage": "出错阶段",
  "traceId": "用于关联服务端日志和 error_report.jsonl 的追踪 ID"
}
```

查看异常报告：

```bash
cat collusive_workdir/batch_75689/output/error_report.jsonl
```

Windows PowerShell：

```powershell
Get-Content collusive_workdir\batch_75689\output\error_report.jsonl
```

异常兼容规则：

- py/AI 运行时异常、超时、PDF 下载失败等运行失败，接口返回 HTTP `200`。
- 响应体仍保持 `AnalyzeResponse` 结构。
- 当前 `itemCode` 只要发生调用级失败，该检查项下所有公司统一写为 `ERROR`。
- Java 后端可以继续调用后续 `itemCode`，最后汇总公司状态和批次状态。
- 参数错误，例如未知 `itemCode`、`companies` 为空，返回 HTTP `400`，属于调用方请求错误。

## 8. 常见问题

| 问题 | 原因 | 处理 |
| :--- | :--- | :--- |
| `Connection refused` | API 服务未启动 | 先运行 `python collusive_check_api.py` |
| `status=ERROR` | 当前检查项执行异常 | 查看 `error_report.jsonl`，用 `traceId` 关联服务端日志 |
| 所有公司都是 `ERROR` | PDF 下载失败、超时、管线异常 | 检查 `bidFileUrl` 是否可访问，或检查本地文件名是否匹配 |
| `Killed` / `已杀死` | 内存不足 | 降低 worker：`PHASE1_WORKERS=1 PDF_CHUNK_WORKERS=1 OCR_COLLECT_WORKERS=1 PHASE3_WORKERS=1` |
| 重型项很慢 | OCR/SBERT/图片相似耗时 | 先用 `--lightweight` 验证接口，再单独跑重型项 |
| 重复测试结果没变化 | 使用了相同 `batchId` 缓存 | 删除 `collusive_workdir/batch_{batchId}` 后重跑 |

## 9. 推荐测试顺序

第一次部署时建议按这个顺序：

```bash
# 1. 启动服务
python collusive_check_api.py

# 2. 轻量项，验证接口和 PDF 可访问
python scripts/test.py --pdf-dir batch_downloads/75689 --lightweight

# 3. 技术标，验证 OCR/SBERT/图片证据
python scripts/test.py --pdf-dir batch_downloads/75689 --item TECH_BID_SIMILAR

# 4. 商务标，验证技术/商务分离和缓存复用
python scripts/test.py --pdf-dir batch_downloads/75689 --item Business_BID_SIMILAR
```

确认以上都正常后，再运行：

```bash
python scripts/test.py --pdf-dir batch_downloads/75689
```
