# 围标串标 AI 服务运行流程与测试流程

本文档说明当前 API 服务的完整运行链路、部署注意事项、测试方法和常见问题排查。当前服务以 Java 后端按 `itemCode` 枚举逐项调用为设计目标，每次请求只处理一个检测维度，但会携带同一批次的全部投标公司。

## 1. 接口调用模式

当前核心接口：

```text
POST /api/v1/collusive-check/items/analyze
```

调用方式：

```text
一个 itemCode + 同批次全部公司 + PDF URL -> AI 分析 -> 返回当前 itemCode 下每家公司结果
```

不是一次请求返回全部维度，也不是按公司逐个调用。Java 后端如果选择多个检查项，会按顺序多次调用：

```text
FILE_CODE_SIMILAR        请求一次
DOC_AUTHOR_SIMILAR       请求一次
TECH_BID_SIMILAR         请求一次
Business_BID_SIMILAR     请求一次
```

每次返回结构：

```json
{
  "batchId": 1783889309,
  "itemCode": "TECH_BID_SIMILAR",
  "itemName": "技术标雷同",
  "results": []
}
```

`status` 含义：

```text
SUCCESS = 当前检测项通过，未发现风险
FAILED  = 当前检测项不通过，发现风险
ERROR   = 当前检测项执行异常、超时或服务内部错误
```

`FAILED` 不是接口调用失败，而是业务检测命中风险。

## 2. 当前支持的 itemCode

| itemCode | 名称 | 类型 | 是否重型 |
| --- | --- | --- | --- |
| `FILE_CODE_SIMILAR` | 文件码雷同 | PDF 指纹 | 否 |
| `EDITOR_SIGNER_SIMILAR` | 编辑经办人雷同 | PDF 元数据 | 否 |
| `DOC_AUTHOR_SIMILAR` | 文档作者雷同 | PDF 元数据 | 否 |
| `SAME_BID_CONTACT_SIMILAR` | 人名雷同 | 正文联系人提取 | 否 |
| `SAME_bidderName_SIMILAR` | 公司名雷同 | 正文公司名提取 | 否 |
| `TECH_BID_SIMILAR` | 技术标雷同 | 文本 + 图片 + OCR | 是 |
| `Business_BID_SIMILAR` | 商务标雷同 | 文本 + 图片 + OCR | 是 |

商务标标准编码现在统一为：

```text
Business_BID_SIMILAR
```

历史编码 `BID_COMPANY_NAME_ABNORMAL` 和 `COM_BID_SIMILAR` 不再作为正式接口编码使用。

## 3. 重型维度运行流程

`TECH_BID_SIMILAR` 和 `Business_BID_SIMILAR` 会走完整管线。

```text
请求进入 API
  -> 下载或复用 PDF
  -> 复制到 batch 工作目录
  -> Phase 0: 扫描 PDF 与元数据
  -> Phase 1: 文本、段落、MinHash、图片哈希、OCR 提取
  -> Phase 1.2: 技术标/商务标页级分类
  -> Phase 1.5: SBERT 段落 embedding 编码与缓存
  -> Phase 2: 候选公司对选择
  -> Phase 3: 段落精细匹配与图片证据构建
  -> Phase 4: JSON 报告对象生成
  -> Phase 5: 输出 detection_report.json
  -> 当前 itemCode 的公司级 JSON 响应
```

技术标和商务标分开检测：

```text
TECH_BID_SIMILAR     -> 只使用 technical 页面内容
Business_BID_SIMILAR -> 只使用 commercial 页面内容
```

如果页级分类失败，严格模式下不会回退到全文混用，避免技术标和商务标互相污染。

## 4. 缓存复用规则

同一个 `batchId` 会复用已有中间结果，减少重复计算。

主要缓存：

| 缓存 | 作用 |
| --- | --- |
| `batch_downloads/{batchId}` | PDF 下载缓存 |
| `collusive_workdir/batch_{batchId}` | 当前 batch 工作目录 |
| SQLite cache | 文档、段落、图片 OCR、候选对、结果缓存 |
| `shared_embedding_cache` | 全局 text_hash embedding 缓存 |
| checkpoint | Phase 进度缓存 |

推荐调用顺序：

```text
先跑轻量项 -> 预热文本缓存
再跑 TECH_BID_SIMILAR -> 技术标检测
再跑 Business_BID_SIMILAR -> 复用已有文本/OCR/embedding 缓存
```

注意：

```text
technical 和 commercial 的 Phase 3 checkpoint 分开保存
但底层 PDF 下载、文本提取、OCR、embedding 缓存可复用
```

## 5. OCR 与图片处理策略

当前默认 OCR 引擎：

```text
RapidOCR / ONNX Runtime
```

默认依赖：

```text
rapidocr-onnxruntime>=1.2.0
```

OCR 处理方式：

```text
1. 直接提取完整嵌入图片进行 OCR
2. 小 PNG 碎片不逐个 OCR
3. 矢量图或碎片图先合并区域，再只渲染合并区域 OCR
4. 不做整页渲染 OCR
```

图片哈希处理方式：

```text
1. cluster_drawings() 检测矢量图/碎片图区域
2. get_image_info() 获取图片/图块 bbox
3. 合并相邻 bbox
4. 只渲染合并区域计算 pHash
5. 对直接嵌入图片使用 doc.extract_image(xref) 计算哈希
```

已经取消 `fast/full` 模式，图片哈希默认执行完整逻辑，但仍然不做整页渲染。

## 6. 推荐 CPU 启动方式

安装依赖：

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements-cpu.txt
```

启动服务：

```bash
export OCR_ENGINE=rapidocr
export COLLUSIVE_ENABLE_OCR=1
export COLLUSIVE_ENABLE_IMAGE_ANALYSIS=1
export USE_GPU=0
export SBERT_DEVICE=cpu

export ANALYZE_EXECUTOR_WORKERS=4
export PDF_DOWNLOAD_WORKERS=4

export PHASE1_WORKERS=2
export PDF_CHUNK_WORKERS=2
export OCR_COLLECT_WORKERS=2
export OCR_WORKERS=1
export PHASE3_WORKERS=2

python collusive_check_api.py
```

8 核 CPU 可以尝试：

```bash
export PHASE1_WORKERS=2
export PDF_CHUNK_WORKERS=4
export OCR_COLLECT_WORKERS=4
export PHASE3_WORKERS=4
```

如果服务端显示 `Killed` 或 `已杀死`，说明大概率被 OOM Killer 杀掉，先降低：

```bash
export PHASE1_WORKERS=1
export PDF_CHUNK_WORKERS=1
export OCR_COLLECT_WORKERS=1
export PHASE3_WORKERS=1
```

## 7. 推荐 GPU 启动方式

GPU 服务器可以用于 SBERT 加速：

```bash
export OCR_ENGINE=rapidocr
export COLLUSIVE_ENABLE_OCR=1
export COLLUSIVE_ENABLE_IMAGE_ANALYSIS=1
export USE_GPU=1
export SBERT_DEVICE=cuda

export PHASE1_WORKERS=2
export PDF_CHUNK_WORKERS=2
export OCR_COLLECT_WORKERS=2
export OCR_WORKERS=1
export PHASE3_WORKERS=4

python collusive_check_api.py
```

说明：

```text
RapidOCR 默认使用 CPU ONNX Runtime。
SBERT 使用 cuda。
如需 ONNX Runtime GPU，需要额外安装 onnxruntime-gpu，但当前默认不依赖。
```

## 9. 如何查看测试结果

测试脚本终端会显示：

```text
[TECH_BID_SIMILAR] status=200 elapsed=xxx.xx s
results=2, failed=0, errors=0
Timing summary
```

重点看：

```text
status=200  -> HTTP 请求成功
errors=0    -> 没有程序异常
failed=N    -> N 家公司业务检测不通过
elapsed     -> 当前 itemCode 总耗时
```

服务端日志重点看：

```text
PDF 下载并发
配置: workers(P1=..., P3=..., PDF_CHUNK=..., OCR_COLLECT=..., OCR=...)
Phase 1 并行模式
[Worker] chunk 并行
OCR: 页级任务收集并行
OCR: xxx.pdf — N 张图片待识别
Phase 1.5: 全局 SBERT 嵌入编码
Phase 3: 精细分析
性能埋点
```

输出 JSON 报告：

```text
collusive_workdir/batch_{batchId}_{dimension}/output/detection_report.json
```

接口响应只返回当前 `itemCode` 对应的公司级结果。

## 10. 常见问题

### 10.1 Connection refused

错误：

```text
ConnectionRefusedError: [Errno 111] Connection refused
```

含义：

```text
测试脚本连不上 http://127.0.0.1:8001
```

检查：

```bash
ps -ef | grep collusive_check_api
ss -lntp | grep 8001
```

如果服务端显示 `Killed` 或 `已杀死`，通常是 OOM。

### 10.2 服务端显示 Killed / 已杀死

常见原因：

```text
1. PaddleOCR 被加载，内存过高
2. worker 数太高
3. PDF 太大，chunk 并行和 OCR 收集并行叠加导致内存峰值过高
```

处理：

```bash
export OCR_ENGINE=rapidocr
export PHASE1_WORKERS=1
export PDF_CHUNK_WORKERS=1
export OCR_COLLECT_WORKERS=1
export PHASE3_WORKERS=1
```

确认安装：

```bash
python -c "from rapidocr_onnxruntime import RapidOCR; print('rapidocr ok')"
```

### 10.3 为什么 status 是 FAILED

`FAILED` 表示业务检测不通过，不表示接口失败。

```text
SUCCESS = 未发现风险
FAILED  = 发现风险
ERROR   = 程序异常
```

### 10.4 similarity 字段含义

技术标/商务标 evidence 中有两层相似度：

```text
similarParagraphs[].similarity
  -> 两家公司在当前维度下的整体文本相似度

paragraphMatches[].similarity
  -> 单个段落对的相似度
```

单个模板句相似度高，不代表整份文件高度相似。后续应结合模板语过滤、有效段落数量和整体相似度共同判定。

### 10.5 模板语言误判

类似：

```text
本项目施工方案采用...
与本投标有关的一切正式往来信函请寄：
```

属于投标模板常见语句，不应作为强证据。建议处理策略：

```text
1. 模板语硬过滤
2. 有效信息词过滤
3. 批次内高频段落过滤
4. FAILED 判定增加最小有效证据数量门槛
```

当前代码已有模板语软过滤，后续可进一步升级为硬过滤。
